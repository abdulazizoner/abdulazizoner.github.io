Overview

1.2 Static File Strategy

Django's static file pipeline has two distinct modes that must both be configured correctly.

Development (DEBUG=True)

Django's built-in django.contrib.staticfiles serves files from each app's static/ subdirectory automatically. No Nginx needed. Files are referenced in templates using the {% load static %} tag.

Production (DEBUG=False)

Django's dev server does not serve static files in production mode. Two things must happen:

  1. python manage.py collectstatic copies all static files from every app into STATIC_ROOT (the staticfiles/ directory).
  2. Whitenoise middleware intercepts static file requests before they reach Django's view layer, serving them directly from STATIC_ROOT — making Nginx optional for static files at the MVP stage.

Settings additions required in base.py

python · aonware_ai/settings/base.py 
# --- Static files ---
STATIC_URL  = '/static/'
STATIC_ROOT = BASE_DIR / 'staticfiles'   # collectstatic target
STATICFILES_DIRS = [BASE_DIR / 'core/static']  # source lookup

# Whitenoise — add AFTER SecurityMiddleware
MIDDLEWARE = [
    'django.middleware.security.SecurityMiddleware',
    'whitenoise.middleware.WhiteNoiseMiddleware',  # ← HERE
    'django.contrib.sessions.middleware.SessionMiddleware',
    # ... rest unchanged
]

# Whitenoise compression + caching
STORAGES = {
    "staticfiles": {
        "BACKEND": "whitenoise.storage.CompressedManifestStaticFilesStorage",
    },
}

⚠ Critical

CompressedManifestStaticFilesStorage appends a content hash to every static filename (e.g. base.abc123.js). The {% static %} tag resolves this automatically. Never hardcode static file paths in templates.

// Key Insight

The window.DJANGO_CONTEXT block is the bridge between the Django template layer and vanilla JS. The JS reads window.DJANGO_CONTEXT.apiBase instead of a hardcoded URL, and window.DJANGO_CONTEXT.slug instead of URLSearchParams. This means the same base.js works in both standalone (from URL) and Django (from context) modes.

Required update to base.js

javascript · core/static/core/js/base.js (modification)

    post = get_object_or_404(
        Post.objects.prefetch_related('tags'), slug=slug
    )
    return render(request, 'core/blog_detail.html', {'post': post})

def project_detail(request, slug):
    project = get_object_or_404(
        Project.objects.prefetch_related('tech_stack'), slug=slug

2.1 Comment Model

The Comment model belongs in the blog app, as it is semantically a property of Post. It will extend TimeStampedModel for consistency with the existing architecture.

python · blog/models.py (addition) 
class Comment(TimeStampedModel):
    """
    A reader comment attached to a Post.
    Moderated via is_approved flag — only approved comments
    are exposed by the public API.
    """
    post        = models.ForeignKey(
                      Post,
                      on_delete=models.CASCADE,
                      related_name='comments'
                  )
    author_name = models.CharField(max_length=120)
    email       = models.EmailField(blank=True, null=True)  # never exposed in API
    content     = models.TextField(max_length=2000)
    is_approved = models.BooleanField(default=False)        # Admin approves

    class Meta:
        ordering = ['created_at']   # chronological, oldest first
        indexes  = [
            models.Index(fields=['post', 'is_approved']),  # fast approved lookup
        ]

    def __str__(self):
        return f"{self.author_name} on '{self.post.title}'"

Post (blog_post) │ id: UUID [PK] │ title: CharField │ slug: AutoSlugField │ content_markdown: TextField │ category: FK → Category │ created_at / updated_at │ │ 1 : N │ ▼ Comment (blog_comment) id: UUID [PK] ← from TimeStampedModel post: FK → Post ← CASCADE delete author_name: CharField(120) email: EmailField (nullable) ← private, not in API response content: TextField(2000) is_approved: BooleanField ← False by default created_at: DateTimeField updated_at: DateTimeField

⚠ Privacy Note

The email field is stored for admin contact purposes only. It must never appear in any API response schema. The CommentOut schema (below) explicitly excludes it.

2.2 Django Ninja Schema

Three schemas are needed: CommentIn (POST body), CommentOut (GET response), and an inline validation layer.

python · api_gateway/api/blog.py (additions) 
from pydantic import field_validator, Field
from blog.models import Post, Comment
import bleach  # poetry add bleach

class CommentIn(Schema):
    """Payload accepted from the client."""
    author_name: str = Field(..., min_length=1, max_length=120)
    email:       str | None = Field(None, max_length=254)
    content:     str = Field(..., min_length=5, max_length=2000)

    @field_validator('content')
    @classmethod
    def sanitize_content(cls, v: str) -> str:
        # Strip all HTML — no XSS via comment content
        return bleach.clean(v, tags=[], strip=True).strip()

    @field_validator('author_name')
    @classmethod
    def sanitize_name(cls, v: str) -> str:
        return bleach.clean(v, tags=[], strip=True).strip()


class CommentOut(Schema):
    """Shape returned to the client. Email is intentionally absent."""
    id:          UUID
    author_name: str
    content:     str
    created_at:  datetime

@router.get("/posts/{slug}/comments", response=List[CommentOut])
def list_comments(request, slug: str):
    post = get_object_or_404(Post, slug=slug)

@router.post("/posts/{slug}/comments",
           response={201: CommentOut},
           summary="Submit a comment (pending moderation)")
def create_comment(request, slug: str, payload: CommentIn):
    post = get_object_or_404(Post, slug=slug)
    comment = Comment.objects.create(
        post=post,
        author_name=payload.author_name,
        email=payload.email,
        content=payload.content,
        is_approved=False,   # always starts unapproved
    )
    return 201, comment

2.4 Validation Layer Summary

Validation happens at three levels, providing defense in depth:

LevelWhereWhat it catches
1 — Schema Pydantic / CommentIn Type errors, length violations, missing required fields
2 — Sanitization field_validator + bleach XSS via HTML injection in author_name and content
3 — Moderation is_approved=False default Spam, offensive content — reviewed manually in Django Admin

PHASE 03

Scale & UX — Pagination

Add pagination to projects and posts endpoints. Entry: Phase 2 complete. Exit: scrolling the index page loads more items without full page refresh.

3.1 Pagination Strategy

We will use cursor-based pagination via Django Ninja's built-in paginate decorator. This is preferred over offset pagination for the following reasons:

PropertyCursor (chosen)Offset (rejected)
Stability ✓ Stable — uses created_at as cursor anchor ✗ Unstable — new rows shift pages
Performance at scale ✓ O(1) — index seek ✗ O(n) — full table scan with OFFSET
UX pattern ✓ "Load more" / infinite scroll ✓ Numbered pages (not needed here)
Complexity ⚠ Slightly more backend code ✓ Simpler

Page size will be 9 items for projects (fills 3-column grid exactly) and 10 items for blog posts.

3.2 Backend Implementation

python · api_gateway/api/portfolio.py (updated) 
from ninja import Router, Schema
from ninja.pagination import paginate, PageNumberPagination

router = Router(tags=["Portfolio"])

class ProjectOut(Schema):
    id:          UUID
    title:       str
    slug:        str
    description: str
    created_at:  datetime

@router.get("/projects", response=List[ProjectOut])
@paginate(PageNumberPagination, page_size=9)
def list_projects(request, **kwargs):
    return Project.objects.all().order_by('-created_at')

The paginated response shape from Ninja becomes:

json · GET /api/v1/portfolio/projects?page=1

⚠ Breaking Change

Adding @paginate changes the response shape from Array to Object. The frontend JS must be updated simultaneously. The existing standalone HTML files use const projects = await res.json() — this will need to become const { items: projects, count } = await res.json(). Plan both changes together to avoid a broken intermediate state.

3.3 Frontend JS Pattern

The "Load More" pattern is the correct UX choice here — it is non-disruptive, works without JavaScript history API complexity, and is straightforward to implement.

javascript · pagination logic (vanilla JS) 
const state = {
  projectsPage: 1,
  projectsTotal: 0,
  projectsLoading: false,
};

async function fetchProjects(page = 1) {
  if (state.projectsLoading) return;
  state.projectsLoading = true;

  const res  = await fetch(`${AO.API}/portfolio/projects?page=${page}`);
  const data = await res.json();

  // data = { items: [...], count: N, page: N }
  const { items, count } = data;

  state.projectsTotal   = count;
  state.projectsPage    = page;
  state.projectsLoading = false;

  renderProjectCards(items, page > 1); // append if page > 1

  // Show/hide "Load More" button
  const shown = page * 9;
  const btn   = document.getElementById('load-more-projects');
  btn.classList.toggle('hidden', shown >= count);
}

// "Load More" button onclick:
document.getElementById('load-more-projects')
  .addEventListener('click', () => {
    fetchProjects(state.projectsPage + 1);
  });

4.1 Full Stack Overview

Each layer has a specific, non-overlapping responsibility. Understanding this separation is essential before configuration begins.

NGINX (Layer 1 — Public Edge) Role: SSL/TLS termination · domain routing · gzip · static file serving Sees: Raw TCP connections from the internet Speaks: HTTP/HTTPS → upstream socket ↓ HTTP (Unix socket or TCP localhost:8000) GUNICORN (Layer 2 — WSGI App Server) Role: Manages multiple Django worker processes Sees: HTTP requests forwarded by Nginx Speaks: WSGI protocol → Django app ↓ Python function calls (WSGI) DJANGO (Layer 3 — Application) Role: URL routing · views · ORM · API (Ninja) · template rendering Sees: Parsed HTTP request objects Speaks: Python objects → Gunicorn ↓ SQL POSTGRESQL (Layer 4 — Data) Role: Persistent storage for all models

Where Whitenoise fits

Whitenoise inserts itself between Nginx and Django as a WSGI middleware. When a request comes in for a static file (e.g. /static/core/css/base.abc123.css), Whitenoise intercepts it and responds directly — Django's view layer never runs. This means static files are served correctly even if Nginx is not configured to serve them (valuable for platforms like Railway, Heroku, Render).

// Performance Note

In high-traffic production, Nginx should serve static files directly (bypassing Gunicorn entirely) for maximum throughput. Whitenoise handles the fallback for platforms where Nginx isn't configurable. Both approaches can coexist — Nginx serves /static/ when it can; Whitenoise handles edge cases.

4.3 Gunicorn Configuration

Gunicorn should be configured via a gunicorn.conf.py file (not CLI flags) for reproducibility. Worker count follows the standard formula: 2 × CPU cores + 1.

python · gunicorn.conf.py (project root)

✓ Zero Downtime

systemctl reload sends SIGHUP to Gunicorn, which performs a graceful restart — in-flight requests complete before old workers are replaced by new ones. Never use systemctl restart in production, which kills all workers immediately.

4.4 Nginx Configuration

Nginx sits in front of Gunicorn and handles SSL termination, HTTP→HTTPS redirect, gzip, and direct static file serving.

nginx · /etc/nginx/sites-available/aonware.ai 
# HTTP → HTTPS redirect
server {
    listen 80;
    server_name aonware.ai www.aonware.ai;
    return 301 https://$host$request_uri;
}

# HTTPS main server block
server {
    listen 443 ssl http2;
    server_name aonware.ai www.aonware.ai;

    # SSL — obtained via Certbot (Let's Encrypt)
    ssl_certificate     /etc/letsencrypt/live/aonware.ai/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/aonware.ai/privkey.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;

    # Gzip compression
    gzip on;
    gzip_types text/css application/javascript application/json;

    # Static files — served directly by Nginx (fastest path)
    location /static/ {
        alias /srv/aonware/staticfiles/;
        expires 1y;
        add_header Cache-Control "public, immutable";
    }

    # All other requests → Gunicorn via Unix socket
    location / {
        proxy_pass         http://unix:/run/aonware/gunicorn.sock;
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
        proxy_read_timeout 30s;
    }
}
