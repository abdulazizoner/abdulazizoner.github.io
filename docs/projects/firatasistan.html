FıratAsistan

A university-focused AI assistant concept that helps students explore answers from educational and institutional documents through retrieval-augmented generation, vector search, and source-grounded chat design.

Overview

FıratAsistan is an ongoing personal project exploring how a RAG-based assistant could help students find information in scattered educational documents — syllabi, regulations, guides, and departmental notices. I am a student at Fırat University studying AI & Data Engineering, and this project grew from a practical question: how can retrieval workflows make document-heavy university information easier to query without relying on generic chatbot answers?

The concept centers on document ingestion, Turkish content handling, vector indexing with Chroma-like storage, and answer generation that references retrieved source sections. I am learning how to design the pipeline end to end: from PDF/text ingestion through chunking, embedding, retrieval, and a simple chat frontend connected to a backend API.

This is not an official Fırat University product. It has not been deployed by the university, has no institutional approval, and has no real users at scale. I present it honestly as a student learning project that explores production-readiness challenges — accuracy, privacy, and reliability — without overclaiming deployment or adoption.

Problem

Students often need information spread across many documents with different formats, update cycles, and writing styles. Searching manually is slow, and generic LLM chatbots can hallucinate answers or ignore the specific institutional context students need. A useful assistant should retrieve relevant document sections first and ground responses in that material, so users can verify answers against original sources.

Goals

  • Build a document ingestion pipeline for educational and institutional text (PDF, plain text)
  • Handle Turkish content in preprocessing, chunking, and retrieval workflows
  • Implement vector indexing and semantic search using Chroma-like storage concepts
  • Design a RAG flow that passes retrieved chunks to an LLM with citation-style references
  • Connect a simple backend API to a chat frontend for interactive querying
  • Think through KVKK-aware data scope, permissions, and responsible deployment constraints

System / workflow

Document upload (PDF / text files) → Text extraction & Turkish preprocessing → Chunking with metadata (doc name, section, page) → Embedding generation → Store in Chroma-like vector database → Student query via chat frontend → Backend API receives query → Embed query → retrieve top-k relevant chunks → Build grounded prompt with source context → LLM generates answer with references → Display response + source citations to user

Technical focus

  • RAG / retrieval-augmented generation architecture
  • Document ingestion, text extraction, and chunking strategies
  • Vector indexing and semantic retrieval (Chroma-like storage)
  • Turkish content handling in preprocessing and embedding quality
  • Backend API design connecting retrieval service to chat frontend
  • Testing, observability, and deployment planning as improvement areas

What I am building

  • Python ingestion scripts for PDF and text document processing
  • Chunking pipeline with metadata tags for document source tracking
  • Vector store setup and embedding workflow for semantic search
  • Backend retrieval API that returns ranked chunks for a given query
  • Prototype chat interface for querying ingested documents
  • Documentation on architecture, known gaps, and privacy considerations

Current state

The project is in active development as a personal learning effort adjacent to my university studies. Core ingestion and retrieval components are being built incrementally. There is no official deployment by Fırat University, no confirmed institutional endorsement, and no claim of real users or production traffic. Answer accuracy is not guaranteed — responses can be incomplete, outdated, or incorrect when retrieval misses relevant chunks or the LLM misinterprets context. Users should always verify against original documents.

Limitations and risk awareness

Document coverage is limited to what has been manually ingested — the assistant cannot answer questions about material not in the index. Turkish NLP quality depends on preprocessing choices and embedding model behavior, which I am still evaluating. Hallucination remains a risk even with RAG: the model may add details not present in retrieved chunks. There is no formal KVKK compliance review, no access control system for sensitive documents, and no unsupported operational metrics or support guarantees. This must not be treated as an authoritative university information source.

Next steps

  • Improve PDF extraction quality and handle more document layout types
  • Tune chunk size and metadata for better retrieval on Turkish academic text
  • Add clearer citation formatting in chat responses
  • Build a small test query set and track retrieval relevance manually
  • Explore basic logging and error handling in the backend API
  • Document deployment options and privacy scope assumptions before any wider testing

FıratAsistan

Öğrencilerin eğitim ve kurumsal belgelerden yanıt aramasına yardımcı olmayı hedefleyen üniversite odaklı YZ asistan konsepti — retrieval-augmented generation, vektör arama ve kaynak temelli sohbet tasarımıyla.

Genel bakış

FıratAsistan, RAG tabanlı bir asistanın öğrencilere dağınık eğitim belgelerinde — ders programları, yönetmelikler, kılavuzlar ve bölüm duyuruları — bilgi bulmada nasıl yardımcı olabileceğini keşfeden devam eden kişisel bir projedir. Fırat Üniversitesi'nde Yapay Zeka ve Veri Mühendisliği okuyorum; proje pratik bir sorudan doğdu: retrieval iş akışları, genel sohbet botu yanıtlarına bağımlı kalmadan belge yoğun üniversite bilgisini sorgulanabilir kılabilir mi?

Konsept belge alma, Türkçe içerik işleme, Chroma benzeri depolama ile vektör indeksleme ve alınan kaynak bölümlerine referans veren yanıt üretimi etrafında dönüyor. PDF/metin almadan parçalama, embedding, retrieval ve basit bir sohbet arayüzünün backend API'ye bağlanmasına kadar hattı uçtan uca tasarlamayı öğreniyorum.

Bu resmi bir Fırat Üniversitesi ürünü değil. Üniversite tarafından dağıtılmadı, kurumsal onayı yok ve ölçekte gerçek kullanıcı iddiası yok. Dağıtım veya benimsenme iddiası olmadan doğruluk, gizlilik ve güvenilirlik gibi üretime hazırlık zorluklarını keşfeden dürüst bir öğrenci öğrenme projesi olarak sunuyorum.

Problem

Öğrenciler genellikle farklı biçimlerde, güncelleme döngülerinde ve yazım stillerinde birçok belgeye yayılmış bilgiye ihtiyaç duyar. Manuel arama yavaştır; genel LLM sohbet botları halüsinasyon üretebilir veya öğrencilerin ihtiyaç duyduğu kurumsal bağlamı yok sayabilir. Yararlı bir asistan önce ilgili belge bölümlerini getirmeli ve yanıtları bu materyale dayandırmalı; böylece kullanıcılar cevapları orijinal kaynaklara karşı doğrulayabilir.

Hedefler

  • Eğitim ve kurumsal metinler (PDF, düz metin) için belge alma hattı kurmak
  • Ön işleme, parçalama ve retrieval iş akışlarında Türkçe içeriği ele almak
  • Chroma benzeri depolama kavramlarıyla vektör indeksleme ve anlamsal arama uygulamak
  • Alınan parçaları atıf tarzı referanslarla LLM'e ileten RAG akışı tasarlamak
  • Etkileşimli sorgulama için basit backend API'yi sohbet arayüzüne bağlamak
  • KVKK odaklı veri kapsamı, izinler ve sorumlu dağıtım kısıtlarını düşünmek

Sistem / iş akışı

Belge yükleme (PDF / metin dosyaları) → Metin çıkarma ve Türkçe ön işleme → Metadata ile parçalama (belge adı, bölüm, sayfa) → Embedding üretimi → Chroma benzeri vektör veritabanına kayıt → Öğrenci sorgusu (sohbet arayüzü) → Backend API sorguyu alır → Sorgu embedding → en iyi k ilgili parçayı getir → Kaynak bağlamıyla dayanaklı prompt oluştur → LLM referanslarla yanıt üretir → Yanıt + kaynak atıfları kullanıcıya gösterilir

Teknik odak

  • RAG / retrieval-augmented generation mimarisi
  • Belge alma, metin çıkarma ve parçalama stratejileri
  • Vektör indeksleme ve anlamsal retrieval (Chroma benzeri depolama)
  • Ön işleme ve embedding kalitesinde Türkçe içerik işleme
  • Retrieval servisini sohbet arayüzüne bağlayan backend API tasarımı
  • Test, gözlemlenebilirlik ve dağıtım planlaması iyileştirme alanları olarak

Geliştirdiklerim

  • PDF ve metin belge işleme için Python alma betikleri
  • Belge kaynağı takibi için metadata etiketli parçalama hattı
  • Anlamsal arama için vektör deposu kurulumu ve embedding iş akışı
  • Verilen sorgu için sıralı parçalar döndüren backend retrieval API
  • Alınan belgeleri sorgulamak için prototip sohbet arayüzü
  • Mimari, bilinen boşluklar ve gizlilik değerlendirmeleri dokümantasyonu

Güncel durum

Proje üniversite çalışmalarıma bitişik kişisel bir öğrenme çabası olarak aktif geliştirmede. Temel alma ve retrieval bileşenleri aşamalı olarak inşa ediliyor. Fırat Üniversitesi tarafından resmi dağıtım yok, onaylanmış kurumsal destek yok ve gerçek kullanıcı veya üretim trafiği iddiası yok. Yanıt doğruluğu garanti değil — retrieval ilgili parçaları kaçırdığında veya LLM bağlamı yanlış yorumladığında yanıtlar eksik, güncel değil veya hatalı olabilir. Kullanıcılar her zaman orijinal belgelere karşı doğrulamalı.

Sınırlamalar ve risk farkındalığı

Belge kapsamı manuel olarak alınan materyalle sınırlı — asistan indekste olmayan konular hakkında yanıt veremez. Türkçe NLP kalitesi ön işleme seçimlerine ve embedding model davranışına bağlı; hâlâ değerlendiriyorum. RAG olsa bile halüsinasyon riski sürer: model alınan parçalarda olmayan ayrıntılar ekleyebilir. Resmi KVKK uyumluluk incelemesi, hassas belgeler için erişim kontrol sistemi veya çalışma süresi/destek garantisi yok. Resmi üniversite bilgi kaynağı olarak ele alınmamalı.

Sonraki adımlar

  • PDF çıkarma kalitesini artırmak ve daha fazla belge düzeni türünü ele almak
  • Türkçe akademik metinde daha iyi retrieval için parça boyutu ve metadata ayarı
  • Sohbet yanıtlarında daha net atıf biçimlendirmesi eklemek
  • Küçük test sorgu seti oluşturup retrieval alakasını manuel takip etmek
  • Backend API'de temel günlük kaydı ve hata yönetimi keşfetmek
  • Daha geniş test öncesinde dağıtım seçeneklerini ve gizlilik kapsamı varsayımlarını belgelemek
