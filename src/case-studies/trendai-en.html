TrendAI

A personal AI trend intelligence workspace for collecting, organizing, and querying information about AI models, tools, and industry developments through retrieval-augmented research workflows.

Overview

TrendAI started as a learning project to understand how data ingestion, metadata design, and retrieval-augmented generation (RAG) can work together in a practical research tool. The AI landscape changes quickly — new models, frameworks, and product announcements appear across blogs, papers, GitHub, and social feeds. I wanted a structured way to capture what I read and ask questions against that material instead of relying on memory or generic chatbot answers.

The project is intentionally scoped as a personal workspace, not a commercial product. I am building it to practice end-to-end skills: collecting sources, normalizing content, attaching metadata, indexing for retrieval, and connecting a chat interface that grounds responses in ingested documents. Gemini API is the primary generation backend I am exploring for query handling and summarization.

At this stage, TrendAI is exploratory and incomplete. It does not claim to predict market trends, replace professional research, or provide real-time intelligence at production quality. It is a student project focused on learning how these systems are designed and where they break down.

Problem

AI research is fragmented across many sources with inconsistent formats and update cycles. Bookmarks, notes, and ad-hoc searches do not scale when you want to compare tools, track model releases, or revisit earlier readings with context. Generic LLM chat can answer fluently but often lacks grounding in your own collected material, which makes verification harder.

Goals

  • Build a repeatable ingestion pipeline for AI-related articles, documentation, and release notes
  • Design a metadata schema that supports filtering by topic, date, source type, and tool category
  • Implement chunking and vector indexing for semantic retrieval over ingested content
  • Connect a RAG chat flow that retrieves relevant chunks before calling the Gemini API
  • Document retrieval quality issues and experiment with basic evaluation approaches
  • Keep the scope honest: learning project, not a deployed production service

System / workflow

Source input (URLs, notes, docs) → Fetch & parse content (Python scripts) → Clean text & extract metadata (title, date, tags, source URL) → Chunk documents with overlap strategy → Generate embeddings & store in vector index → User query → Embed query → retrieve top-k chunks → Build prompt with retrieved context → Gemini API generation → Response with source references (when available)

Technical focus

  • Python-based ingestion scripts and text preprocessing
  • Metadata organization and source provenance tracking
  • Document chunking strategies and overlap tuning
  • Vector retrieval concepts (embedding models, similarity search)
  • Gemini API integration for grounded generation
  • Basic retrieval evaluation (manual spot-checks, relevance notes)

What I am building

  • Scripts to ingest and normalize content from selected AI news and documentation sources
  • A lightweight metadata layer for tagging models, tools, and trend categories
  • Chunk indexing workflow with vector storage for semantic search
  • A prototype RAG chat interface that passes retrieved context to Gemini
  • Notes and experiments on retrieval quality, chunk size, and prompt structure
  • Project documentation describing architecture decisions and known gaps

Current state

The project is ongoing and at an early prototyping stage. Core ingestion and organization ideas are in place, and I am iteratively connecting retrieval and chat components. There is no public deployment, no user base, and no SLA or unsupported operational metrics. Output quality varies depending on source freshness, chunking choices, and retrieval accuracy. I treat every answer as something that still needs manual verification against original sources.

Limitations and risk awareness

Source freshness is a major constraint: ingested content can become outdated quickly, and the pipeline does not yet automate timely re-fetching for all sources. LLM responses can hallucinate or misinterpret retrieved context, especially when chunks are incomplete or ambiguous. Retrieval evaluation is still informal — I do not have rigorous benchmark metrics yet. This project does not predict market movements, investment outcomes, or competitive positioning. It is a research and learning tool, not financial or strategic advice.

Next steps

  • Improve ingestion reliability and add clearer source timestamps
  • Experiment with chunk size, overlap, and metadata filters for better retrieval
  • Define a small set of test queries and track retrieval hit rate manually
  • Add clearer citation formatting in chat responses
  • Explore scheduled re-ingestion for high-priority sources
  • Document failure cases (missed retrieval, stale data, hallucination) for learning
